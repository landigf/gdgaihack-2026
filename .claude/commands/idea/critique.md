# /idea:critique

Follow this workflow:
1. Read the brainstorm and spec docs before critiquing.
2. Score the idea on user value, technical feasibility, demo impact, and implementation risk.
3. Add feedback to `05-feedback.md` if a feature folder already exists; otherwise include a standalone critique.
4. Keep the output constructive and specific enough for non-CS teammates to act on it.

If a required document is missing, create it from the matching template in `doc/specs/_templates/`.
