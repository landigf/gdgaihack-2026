## Brief vs picked direction

Citation convention: "Brief paragraph N" counts the captured brief block in `02-specification.md` from the first tagline paragraph through the judging-criteria rows.

Control-Room Incident Copilot should be confirmed as the tentative pick: the brief asks for AI applications whose intelligence runs locally, connects to desktop software or the filesystem, and proves useful without Wi-Fi, which maps directly to a control-room workstation that retrieves local SOPs, opens source files, writes an incident log, and shows zero egress during the demo (Brief paragraphs 2, 3, 4, 14, 31, 35). The pick is stronger than the older field-wearable framing because the brief explicitly rewards integration with the local computer ecosystem and penalizes isolated local chatbots, so the visible stage action must be MCP/file/tool use, not just an offline answer box (Brief paragraphs 20, 31, 35). The hardware-first story also fits: the brief rewards optimization for the machine we have, not raw model size, so the pitch should lead with a measured 4B-class model, quantization, memory ceiling, latency, and benchmark delta on the 18 GB M3 Pro while bridging to MSI AI Edge PC deployment (Brief paragraphs 4, 15, 16, 17, 44).

## Risks the brief introduces

- **Risk: judges read the product as "chatbot but local."** Mitigation: make the first demo output call local tools: search SOPs, open the cited PDF/page in Preview, and write the incident JSON to disk, with a visible MCP/audit log on screen (Brief paragraphs 20, 31, 35, 46).
- **Risk: one stray cloud AI call disqualifies the track submission.** Mitigation: disable hosted inference, cloud embeddings, cloud STT/TTS/OCR, browser AI, and hybrid fallbacks; run `scripts/netproof.sh` plus `lsof -i` before and during the demo (Brief paragraphs 9, 10, 11, 12, 13, 14, 33, 34, 37).
- **Risk: model ambition exceeds the 18 GB hardware tier.** Mitigation: ship Gemma/Phi/Qwen 4B-class Q4_K_M as the default, keep larger named models as "evaluated but out of budget," and show the memory/latency tradeoff as an optimization decision (Brief paragraphs 15, 16, 17, 18, 44).
- **Risk: sponsor-fit story becomes generic "private AI" instead of MSI-specific systems engineering.** Mitigation: make slide 5 the proof slide: model, quant, tokens/sec, peak RSS, cited-checklist completeness, and zero-egress result on the exact demo machine, then bridge to MSI AI Edge PC / Copilot+ class hardware (Brief paragraphs 4, 16, 17, 44).
- **Risk: the old voice/wearable plan misses the brief's desktop integration requirement.** Mitigation: frame voice as optional input, not the product; the MVP must be a desktop/control-room panel with local RAG, filesystem writes, local app/file opening, and airplane-mode survivability (Brief paragraphs 3, 4, 19, 20, 22, 31, 35).

## Three lines from the existing pitch seeds we should change

- **Seed C hook, replace:** "A control-room operator at an upstream gas plant hears a hissing alarm in pump room B. One worker coughing. Ventilation fault. The site is an ATEX Zone 1 area where consumer phones are not even legal." **With:** "A chemical-plant control-room operator is staring at the EdgeXpert workstation when Pump Room B alarms: chlorine, one worker coughing, ventilation fault, and the WAN is locked down." This makes the workstation the hero and keeps the no-cloud failure mode visible (Brief paragraphs 3, 4, 14, 31, 35).
- **Seed C solution, replace:** "PoliSa is a local voice-first incident copilot; MSI box above ground pairs with an intrinsically-safe handheld. All reasoning stays on-device, cited from API RP 754 + site SOPs." **With:** "PoliSa is an MCP-native incident copilot: local Gemma, local embeddings, local SOP RAG; it opens the cited procedure, writes the incident log, and sends zero packets." This scores Creative On-Device by showing tool/file integration, not only local voice (Brief paragraphs 20, 22, 31, 35, 46).
- **Seed A/B/C benchmark line, replace any paper-only metric such as:** "Pocket RAG: 14.2s -> 3.7s, 94.5% accuracy." **With:** "On this M3 Pro 18 GB: `gemma3:4b` Q4_K_M, tokens/sec, peak RSS, p50 end-to-end, cited-checklist completeness, and zero-egress proof from `latest.md`." The brief's highest-weight axis is technical optimization, so our measured hardware numbers must outrank borrowed research numbers (Brief paragraphs 15, 16, 17, 44).

## Open questions for organizers Q&A

- What exact MSI machine will judges use or expect us to target: RAM, GPU/VRAM, NPU TOPS, OS, and whether OpenVINO/Foundry Local/ONNX Runtime are preinstalled or allowed? This determines the model ceiling and optimization story (Brief paragraphs 4, 15, 16, 17, 18, 29, 44).
- During judging, will the airplane-mode test be performed by organizers live, or should teams self-demonstrate with `tcpdump`/packet logs? This affects how we stage zero-egress proof (Brief paragraphs 14, 32, 37).
- Does "connected to the OS, local software, or filesystem" require visible tool execution on stage, or is local RAG plus local file writes sufficient? This clarifies how much MCP/Open Interpreter work must be in the MVP (Brief paragraphs 3, 20, 31, 35, 46).
- Are local non-AI services such as a local Streamlit/Tauri frontend, SQLite database, localhost Ollama endpoint, and local file watcher acceptable as one-machine components? This clarifies the single-device rule without weakening the local-inference constraint (Brief paragraphs 3, 13, 18, 22, 36).
- Are side-challenge integrations allowed if they are detachable and run after the core offline demo, especially M5Stack or Luxonis local hardware versus cloud-first sponsors? This preserves the main track's no-cloud AI boundary (Brief paragraphs 8, 9, 10, 11, 14, 33, 34).
