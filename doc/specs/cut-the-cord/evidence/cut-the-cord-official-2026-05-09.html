The Challenge

Most AI hackathons follow the same script: grab an API key, wrap a prompt around it, ship a thin UI. This track flips that entirely.

Build AI applications where 100% of the intelligence runs locally on the machine — connected to the OS, local software, or the filesystem to create tangible impact on a real use case.

Your goal is to connect local LLMs, VLMs, or specialized models (e.g., Gemma 3, gpt-oss-20b, MedGemma, Devstral Small 2) to local desktop software, creating solutions that are private by default, zero-latency, always available, and free from API costs.

We provide Google Cloud credits to fuel your training and fine-tuning phases, but the final product runs entirely on-device. You provide the engineering creativity — designing local RAG architectures, orchestrating multi-agent workflows, enabling agent-to-agent communication, or building computer-use agents that interact directly with the OS. The inference runs 100% locally on consumer hardware — proving your solution is ready for next-gen AI machines like the MSI AI Edge PC or Copilot+ Prestige/PRO laptops.

Mandatory Rules

Hard constraints — violation = disqualification
  • Zero Cloud AI at demo time. No API calls to external AI services (OpenAI, Anthropic, Google, Groq, Together, Replicate, etc.) for any AI functionality. If the wifi drops, your AI must still work exactly the same.
  • 100% Local Inference. Everything — from text generation to speech-to-text and text-to-speech, from embeddings to vision — must run entirely on-device.
  • Software Integration. The AI cannot be an isolated terminal chatbot. It must interact with the OS, local files, desktop applications, or local services to solve a real use case.
  • Single machine. All AI inference must run on one physical device. No distributed inference across team members' laptops.
What IS allowed
  • Cloud for training/fine-tuning — Use your Google Cloud credits to train or fine-tune models before the event or during the hackathon. The constraint is about runtime inference at demo time.
  • Downloading models during setup — Pulling models from Hugging Face, Ollama, etc. before your demo is fine.
  • Cloud for non-AI operations — Fetching datasets, scraping web content, hosting a frontend, authentication, databases — all fine. The line is clear: AI inference = local, everything else = your choice.
  • External hardware accelerators — eGPUs, Coral TPUs, Intel Neural Compute Sticks — if it's plugged into your machine, it counts as on-device.
What is NOT allowed
  • Any hosted AI inference API (including "free tier" or "self-hosted on a VPS")
  • Cloud-based embeddings, speech-to-text, text-to-speech, OCR, or vision APIs
  • "Hybrid" approaches where the model falls back to cloud for harder queries
  • Browser-based AI features that phone home to cloud servers
  • Running your model on a remote server and calling it locally
The Airplane Mode Test

The simplest mental model: if we enable airplane mode right before your demo, does your AI still work? If yes, you pass. If no, you fail the track constraint. Constraint compliance is binary — violation means disqualification from track prizes.

Hardware Reality

You bring your own machine. This is part of the challenge.

A team with 96 GB of unified memory has access to larger models than a team with 16 GB — and that's intentional. The judging criteria reward optimization for the hardware you have, not raw model size. A team running a 7B model beautifully on a 16 GB laptop with smart caching and a great UX can outscore a team barely running a 70B model with a broken interface.

Recommended minimum specs:

Tier RAM GPU What you can realistically run
Minimum 16 GB Any dedicated GPU, Apple Silicon, or recent AMD APU 3–8B models (Q4): gpt-oss-20b, Phi-4, Gemma 3 4B, Mistral Small
Comfortable 32 GB 8+ GB VRAM or Apple M2+ 8–14B models (Q4): Llama 3.1 8B, Devstral Small 2, MedGemma 4B
Ideal 64+ GB 16+ GB VRAM or Apple M3/M4 Pro+ 30–70B models (Q4): Llama 3.1 70B, Gemma 3 27B, gpt-oss-120b

Teams below the minimum can still participate but should expect significant model limitations. Plan your architecture around your hardware, not the other way around.

Recommended Open-Source Stack

Inference & Model Serving

Ollama / LM Studio
Drop-in local inference with massive model libraries. Easiest way to get started.
llama.cpp
The gold standard for CPU/GPU quantized inference (GGUF). Maximum control over quantization and performance.
MLX
Apple Silicon optimized. Best local performance on Mac hardware.
TensorRT-LLM / ONNX Runtime
Maximum hardware acceleration on NVIDIA GPUs and Windows NPUs.
vLLM
High-throughput local serving with PagedAttention. Great for multi-model setups.

Voice & Audio (100% Local)

whisper.cpp
Extremely lightweight local speech-to-text. C++ optimized, runs on CPU.
Kokoro-82M
Apache-licensed, 82M-parameter TTS that rivals models 10× its size. Top-ranked on TTS Arena.
Piper
Fast local TTS optimized for Raspberry Pi and edge devices. Multiple voices and languages.

Agentic Frameworks & OS Integration

Model Context Protocol (MCP)
The standard for securely connecting local LLMs to filesystems, databases, IDEs, and desktop apps. Essential for this track.
Google ADK
Google's open-source framework for multi-agent orchestration. Model-agnostic, supports local models via LiteLLM integration.
LangGraph / CrewAI
Advanced orchestration for stateful multi-agent systems and role-based agent teams.
SmolAgents / AutoGen
Lightweight frameworks for building local multi-agent and agent-to-agent workflows.
Open Interpreter
Local computer-use implementation that lets LLMs execute code, manage files, and control the OS with built-in safety layers.
PyAutoGUI / SikuliX
Programmatic OS control for agents that need to interact with desktop application GUIs.

Secure Execution & Sandboxing

Docker
Containerize agent environments and isolate local tool execution. Essential for safe code-generating agents.
E2B (Self-hosted/Desktop)
The industry standard for sandboxed code execution (Code Interpreter) to let agents run Python/JS safely in isolated environments.

Local RAG & Search

ChromaDB / Qdrant / FAISS
Local vector databases. All run entirely on-device.
sentence-transformers / nomic-embed / BGE
Local embedding models for building your RAG pipeline without cloud embedding APIs.

Case Studies

These are aspirational examples designed to inspire, not literal 24-hour blueprints. Take the core idea and scope it to what's buildable in a hackathon.

01Local Multi-Agent Cyber-Defense SOC

The Problem

Modern SIEM/SOAR systems require uploading sensitive telemetry to the cloud, creating privacy risks and latency in responding to threats.

The Vision

A "SOC-on-a-Chip" — a Log Sentinel Agent monitors syscalls and network traffic via MCP; a Threat Hunter Agent performs local RAG on vulnerability databases; an Incident Responder Agent uses OS-level tools to isolate suspicious processes or modify firewall rules. The agents negotiate via agent-to-agent communication to validate threats before taking action. Zero data leaves the device.

02GDPR-Compliant Medical Scribe (Local RAG)

The Problem

Doctors spend hours on clinical documentation, but patient data cannot legally be streamed to public clouds under GDPR.

The Vision

An offline medical assistant. Whisper.cpp transcribes the visit, while a local RAG system queries a secure local patient database. MedGemma 4B generates structured clinical notes — entirely on-device. The privacy advantage isn't a marketing claim, it's an architectural guarantee.

03Offline Accessibility Controller (Agent-to-Agent)

The Problem

Severe motor disabilities make standard PC navigation difficult, and cloud-based voice assistants have latency that makes real-time interaction frustrating.

The Vision

A voice agent and a vision agent working together locally. The voice agent interprets commands via whisper.cpp; the vision agent (local VLM) "sees" the current UI state. They communicate agent-to-agent to execute complex system macros — opening apps, filling forms, navigating menus — with zero latency via the NPU.

04Privacy-First Smart Restaurant Concierge

The Problem

Restaurants handle sensitive customer data (allergies, dietary restrictions, VIP preferences) and face chaos when internet goes down during peak hours.

The Vision

A multi-agent local hub. A Front-of-House Agent (local TTS/STT) manages bookings and dietary records with total privacy. An Inventory Agent connects to the local POS/ERP via MCP to track stock in real-time and runs optimization algorithms in a Docker sandbox to adjust menu recommendations. The agents communicate: the Inventory agent tells the Concierge which dishes to push based on freshness and margins. All 100% offline.

Evaluation Criteria

Criterion What judges evaluate Weight
Technical Optimization Systems engineering quality. Model selection, quantization strategy, memory management, inference speed. How well does the AI perform given your specific hardware? A team that runs Phi-4 at 40 tok/s with smart caching should outscore a team barely running a 70B at 2 tok/s. 30%
Practical Utility Real-world impact. Does this solve an actual problem someone has? Would someone use this beyond the hackathon? 25%
Creative On-Device Use Depth of integration between the local model and the computer's ecosystem. RAG pipelines, multi-agent orchestration, MCP connections, OS-level interaction, creative multi-model combinations. Did you go beyond "chatbot but local"? 25%
Competitive Advantage Clear demonstration of why the local solution is superior to cloud. Privacy guarantees, zero latency, offline reliability, zero API cost. The best projects have use cases where local AI is the correct architecture, not just a constraint. 20%
Total 100%

Getting Started

  1. Pick your domain. The constraint is architectural, not thematic. Productivity, creativity, accessibility, developer tools, healthcare, education, security — any domain works.
  2. Audit your hardware. Check your RAM, GPU VRAM, and whether you have an NPU. This determines your model ceiling.
  3. Set up Ollama or LM Studio. Pull a model that fits your hardware. Test it works before the hackathon.
  4. Think about integration. The "software integration" rule is what separates winning projects from local chatbots. How will your AI connect to the OS, files, or apps?
  5. Design for the demo. Judges may ask you to demo in airplane mode. Make sure your non-AI features degrade gracefully without network.
